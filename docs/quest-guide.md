# Quest Guide — how to author a well-formed quest

> **Audience:** the intake leg (an Opus conversation running the `quest-intake`
> skill) and any human reading a quest manifest. This file is the canonical
> definition of "what a well-formed quest is." The `quest-intake` skill turns
> these rules into a live interview; this guide is the rulebook it enforces.

> Parent design: `.claude/docs/plan-quest-engine.md` (the frozen §-register).
> Schema owner: `src/shared/quest-schema.ts` (packet H2). **Sync status: see the

## 1. What a quest is

A **quest** is one unit of delivery — a goal carried *past a single context
horizon* by many short-context conversations. You do not run a quest in one
sitting; you decompose it into **legs**, each leg a fresh ~2-hour conversation
that executes exactly one task and hands off. The engine dispatches legs,
verifies each one at a hard gate, and computes "quest complete" as a predicate
over the board — never a claim an agent makes about itself.

**Vocabulary (load-bearing — use these exact words):**

| Term | Meaning |
|---|---|
| **Quest** | The whole run: a petname-selected set of board cards + a manifest folder. The unit of delivery. |
| **Leg** | One conversation executing one task. Fresh context, ~2h, ends TERMINAL with a recap. |
| **Baton** | The manifest + append-only log. What a leg reads to know what came before and what it may safely redo. |
| **The Guard** | The verifier. An LLM that runs the acceptance command and *distrusts the worker* ("reject aggressively"). Gated behind deterministic machine checks. |
| **Checkpoint** | A human GO/NO-GO gate. Mandatory before `shipped` (§7c). |
| **Quest Giver** | Jonas. Blesses the intake, holds the kill switch. |
| **Petname** | `floppy-panda` — the autogenerated unique selector. Public, not secret. Human + machine mnemonic. |

The **intake** (this guide's subject) is the HARD GATE at the front: no quest
enters the engine without a blessed manifest whose every task carries a
machine-checkable acceptance command. A vague quest is DECLINED, not softened.

> **Naming collision — don't conflate two "quests".** The broker already ships an
> unrelated `dispatch_quest` desk tool (`src/broker/desk/quest-tool.ts`): an
> *ephemeral, in-memory* "spawn a worker to answer a question and report back"
> verb with no manifest and no `quests/` folder. This guide is about the
> **quest-engine quest** — a filesystem-backed, petname-selected unit of delivery
> (`.rclaude/project/quests/<petname>/`). When you read "quest" here, it is
> always the manifest-backed engine quest, never `dispatch_quest`.

## 2. The manifest schema

A quest lives at `.rclaude/project/quests/<petname>/`:

```
.rclaude/project/quests/floppy-panda/
├── manifest.md      # YAML frontmatter (the schema below) + prose body
├── log.md           # append-only baton — intent/completion/plan/steering entries
└── artifacts/       # graphics, plan docs, steering notes, Guard evidence
```

The **board cards** for the quest's tasks live in the normal lanes
(`.rclaude/project/{inbox,open,in-progress,in-review,done,archived}/`) and stay
**thin** — they carry only a `quest: <petname>` frontmatter key (§4a) that ties
them to the manifest. Membership is orthogonal to lane (§4c): a card is a quest
member because it says `quest: floppy-panda`, regardless of what lane it sits in.

### 2.1 `manifest.md` frontmatter — field by field

| Field | Type | Required | Meaning |
|---|---|---|---|
| `petname` | string | yes | The quest selector, e.g. `floppy-panda`. Generated at intake (§4d), collision-checked against existing quests. This is also the folder name. |
| `project` | string | yes | The project the quest runs against. The `claude://` URI or the plain project name — **informational to the broker**, the sentinel owns URI↔path. |
| `goal` | string | yes | One-sentence delivery statement. What "done" means at the quest level, in plain language. |
| `target` | enum | yes | `pr` \| `merged` \| `shipped`. The delivery rung — rewrites the completion predicate (§7). |
| `status` | enum | yes | `intake` \| `armed` \| `running` \| `paused` \| `complete` \| `aborted`. Lifecycle of the quest itself (distinct from card lanes). Intake writes `armed` on blessing. |
| `gate` | object | yes | The gate verdict. `{ verdict: blessed \| declined \| pending, blessed_by, blessed_at }`. The blessed intake IS the hard gate's output (§1a). |
| `tasks` | list | yes | The decomposition — one entry per leg. Each carries its acceptance contract. See §2.2. |
| `constraints` | list of strings | no | Standing rules every leg must honor (e.g. "no new dependencies", "touch only `src/broker/`"). |
| `deny_floor` | list of strings | no | Extra denies layered on top of the profile's permission floor for this quest's unattended legs (§10 of plan-nightshift). |
| `created` | ISO-8601 | yes | Manifest creation time. |
| `updated` | ISO-8601 | yes | Last `update_quest` patch time. |

### 2.2 `tasks[]` — one entry per leg

| Field | Type | Required | Meaning |
|---|---|---|---|
| `id` | string | yes | Stable task id within the quest, e.g. `t1`. Matches the board card's own id linkage. |
| `title` | string | yes | Imperative one-liner: "Add `--version` flag to broker-cli". |
| `accept` | string | yes | **The machine-checkable acceptance command** (§3). A shell command that exits 0 iff the task is truly done. This is the contract. |
| `accept_note` | string | no | Human gloss of what `accept` proves, for the Guard and the reviewer. |
| `kind` | enum | no | `build` (default) \| `scout` (research, produces a doc not a diff) \| `integrate` (merge/deploy leg for higher targets). |
| `depends_on` | list of ids | no | Task ids that must reach terminal-delivered before this leg dispatches. Feeds the integrator's DAG (§7b). |

### 2.3 `log.md` — the append-only baton

Never rewritten. Only appended (the `quest_log_append` verb is separate from
`update_quest` precisely so the log can NEVER be patched away — §4e). Each entry:

| Field | Meaning |
|---|---|
| `ts` | ISO-8601 timestamp. |
| `kind` | `intent` (what a leg was about to do) \| `completion` (what it delivered, machine-authored git facts + agent narrative) \| `plan` (a replan version) \| `steering` (a Quest-Giver course correction). |
| `convId` | The conversation (leg) that wrote the entry. |
| `body` | Markdown. For `intent`: what I was about to do, worktree state, what's safe to redo (§3 resumer schema). |

### 2.4 Worked example

`.rclaude/project/quests/floppy-panda/manifest.md`:

```markdown
---
petname: floppy-panda
project: claude://sentinel/Users/jonas/projects/remote-claude
goal: Ship a `--version` flag on broker-cli that prints the package version.
target: merged
status: armed
gate:
  verdict: blessed
  blessed_by: jonas
  blessed_at: 2026-07-05T21:14:00Z
tasks:
  - id: t1
    title: Add --version flag to broker-cli
    accept: "cd \"$WORKTREE\" && bun test src/broker/cli/version.test.ts && ./broker-cli --version | grep -Eq '^[0-9]+\\.[0-9]+\\.[0-9]+'"
    accept_note: Unit test covers flag parsing; the grep proves the real binary prints a semver line.
    kind: build
constraints:
  - Touch only src/broker/cli/ and its tests.
  - No new dependencies.
deny_floor:
  - Bash(git push:*)
created: 2026-07-05T21:10:00Z
updated: 2026-07-05T21:14:00Z
---

## Why now

`broker-cli` has no way to report its own version, so ops can't tell which build
a container is running without `docker inspect`. A `--version` flag closes that.

## Notes

Version source is the package.json already bundled into the CLI. No network.
```

`.rclaude/project/quests/floppy-panda/log.md`:

```markdown
### 2026-07-05T21:14:00Z · intent · conv_intake_abc123
Quest blessed at intake. Target `merged`, one build leg (t1). Acceptance is a
unit test + a real-binary grep. Next: dispatch t1 in a fresh worktree.
```

## 3. Acceptance contracts — THE PRIME DIRECTIVE

> **Every task MUST carry a machine-checkable acceptance command authored BEFORE
> work starts. A task that cannot be made concrete is DECLINED, not softened.**

This is the whole point of the gate. The engine runs unattended; the only thing
standing between "an agent said it's done" and reality is a command that a dumb
machine can run and read the exit code of. If you can't write that command, the
task is not ready to dispatch — and the honest outcome is to decline it, not to
wave it through with a fuzzy description.

An acceptance command is well-formed when:

1. **It exits 0 iff the task is genuinely done, non-zero otherwise.** The Guard
   reads the exit code, not prose.
2. **It is runnable in a fresh worktree with no human in the loop.** Assume
   `$WORKTREE` is the leg's checkout. No "open the app and look."
3. **It proves the actual behavior, not a proxy.** "The file exists" is weak;
   "the binary prints a semver line" is strong.
4. **It is deterministic.** No flaky network, no wall-clock, no random ordering.

### Good vs bad contracts

| ❌ Bad (DECLINE or push back) | ✅ Good (accept) |
|---|---|
| "works well" | `bun test src/broker/cli/version.test.ts` exits 0 |
| "the flag is added" | `./broker-cli --version \| grep -Eq '^[0-9]+\.[0-9]+\.[0-9]+'` |
| "tests pass" (which tests?) | `cd "$WORKTREE" && bun test src/broker/foo` exits 0 |
| "the UI looks right" | a Playwright assertion, OR split off a human checkpoint task |
| "refactor is clean" | `bun run lint && bun run tsc` exit 0 AND named tests still green |
| "no regressions" | the specific test list that must stay green, run as one command |

### When a task resists a contract

Push back first. Most "unmeasurable" tasks are actually under-decomposed —
break them until each piece has a command. If a piece is genuinely
research/unknown ("figure out why X is slow"), it is a **Scout task** (§5):
`kind: scout`, and its acceptance is the *artifact* it must produce
(`test -f "$WORKTREE/docs/x-slowness-findings.md"` plus a content check), not a
behavior change. If even that can't be pinned, **decline the task and say so
loudly** — declining is a valid, respectable intake outcome. A soft contract is
worse than no quest: it launders a vague goal into false confidence.

## 4. Target ladder (§7)

Declared once per quest at intake. The target rewrites the completion predicate
and the strictness of the gate.

| Target | Means | Completion predicate | Gate strictness (§7d) |
|---|---|---|---|
| `pr` | Every task green on its branch AND raised as a reviewable PR. No-remote projects: a review card + branch is the PR-equivalent. | Each task's `accept` exits 0 on its branch; PR (or review card) exists. | Baseline: acceptance command + committed + clean tree + nonzero diffstat. |
| `merged` | Integrated to main. Pulls in the integrator: dependency-DAG ordering, cross-worktree conflict handling, ff-merge policy. | All of `pr` + every branch merged to main, main stays green. | Stricter: also requires the integrator leg to pass and no merge conflicts left. |
| `shipped` | Deployed. Deploy + smoke + rollback. | All of `merged` + deploy succeeded + smoke passed. | Strictest. **Crosses the irreversibility line → a MANDATORY human checkpoint sits between `merged` and `shipped` (§7c). Auto-GO is OFF by default here, and only here.** |

**§7d in one line: higher target = stricter §1 gate.** A `shipped` quest demands
tighter acceptance contracts than a `pr` quest, because more of the pipeline runs
without a human. When the interview sets `target: shipped`, tighten every
contract accordingly and confirm the Quest Giver understands the checkpoint.

**Default:** if the Quest Giver doesn't say, propose `merged` — it's the common
case (work that should land on main) and it forces the integrator to be honest
about conflicts. Only go `pr` when review-before-merge is explicitly wanted, and
`shipped` only when deploy is truly in scope.

## 5. Sizing guidance

**One task = one leg = one ~2-hour fresh-context conversation.** That is the
unit. Size every task to fit inside a single leg with room for the Guard.

- **Too big** (won't fit one context, or spans multiple subsystems) → **split**
  it until each piece is a single-leg task with its own acceptance command.
- **Unknown** (you can't yet write a contract because the shape isn't known) →
  put a **Scout task first** (`kind: scout`): a research leg whose output is a
  findings doc. The follow-up build tasks depend on it and get real contracts
  once the Scout returns.
- **Sequential dependencies** → set `depends_on`; the engine orders legs by the
  DAG (matters for `merged`/`shipped` integration).
- **Few sharp reviewable tasks beat many vague ones.** A quest with three tasks
  that each have a crisp `accept` command is worth more than ten mushy ones.
  Ruthless scoping is the interviewer's job.

Rule of thumb: if you can't describe what a leg *finishes with* in one sentence
and check it with one command, it's mis-sized. Split or Scout.
